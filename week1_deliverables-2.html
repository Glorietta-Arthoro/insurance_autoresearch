Week 1 Deliverables — Healthcare Claims Denial Prediction

Binary Classification | STAT 390 Capstone | Agent modifies train.py to maximize F2 score on denied claims

1 Project Charter

The Problem

Healthcare providers submit millions of insurance claims every year and a significant portion come back denied due to coding mismatches, missing authorizations, or eligibility issues. Billing teams must then manually correct and resubmit each one, a process that is slow, expensive, and sometimes never completed. Denial rates commonly run between 10 and 15 percent of all claims, representing millions of dollars in delayed or lost reimbursements. This project builds an autonomous ML pipeline that predicts which claims are likely to be denied before they are ever submitted, so billing teams can fix errors upfront.

What the Agent Optimizes

F2 score on the denied class, measured on the validation set after each experiment. F2 weights recall more heavily than precision, reflecting the real cost asymmetry where a missed denial is far more damaging than a false alarm.

Data Source

Kaggle synthetic healthcare claims dataset. Contains individual claim-level records with procedure codes, diagnosis codes, payer type, claim amount, and a denial_status column (Paid / Denied / Partially Paid) as the binary target variable.

Editable File

train.py — the full modeling pipeline including feature encoding, model selection, class imbalance handling, and decision threshold. The only file the agent may touch.

Frozen Files

prepare.py — data loading, deterministic 70/15/15 split with fixed seed, F2 evaluation function. run.py — experiment runner and logger. Neither file may be modified by the agent under any circumstances.

Baseline

Logistic regression on payer type, claim amount, and frequency-encoded procedure code. Establishes the floor F2 score every agent experiment must beat before being committed to git.

Minimum Viable Project

A reproducible AutoResearch loop completing at least 30 experiments with a clean keep/discard log, and a final model that outperforms the logistic regression baseline on F2 score on the validation set.

2 Research Question

One Clear Research Question

Can an autonomous ML agent, operating within the AutoResearch ratchet loop, discover a claims classification model that predicts insurance denial before submission with an F2 score above 0.70 on held-out test data?

This question is evaluable (single numeric threshold), bounded (one dataset, binary classification), reproducible (deterministic split, full experiment log), and answerable within 8 weeks.

3 Success Criterion

F2 > 0.70
Primary metric on locked test set
F2 weights recall 2x over precision. Chosen because missing a denial costs more than a false alarm.
Recall > 75%
On denied class in test set
The model must correctly flag at least 3 in 4 claims that would truly be denied. Below this the billing team sees no meaningful reduction in denial rate.
Secondary criterion: A rigorous, reproducible pipeline with a complete experiment log and honest failure analysis also constitutes a successful project per course policy, even if the F2 threshold is not met.

5 Risk List

# Risk Severity Mitigation How to Detect
1 Agent modifies frozen files
Agent edits prepare.py or run.py, making experiments incomparable		 High Explicit rule in program.md: never touch prepare.py or run.py. Only train.py is listed as editable. git diff --name-only after each run — only train.py should appear
2 Severe class imbalance stalls improvement
If denials are under 10% the agent may fail to improve recall meaningfully and F2 stagnates		 High Check denial rate in week 2 exploratory analysis. Set class weights explicitly in baseline. Add SMOTE as a directed early experiment in program.md. F2 score below 0.30 after 10 experiments with no upward trend
3 High cardinality encoding explodes feature space
Naive one-hot encoding of CPT and ICD-10 codes causes training to crash or produce noise		 High Set frequency encoding as default in train.py before loop begins. List target encoding and category grouping as directed alternatives in program.md. MemoryError or training time exceeding budget on first few experiments
4 Metric is non-reproducible
Random seed not fixed — same model gives different F2 on different runs		 Medium RANDOM_SEED = 42 frozen in prepare.py. Deterministic split. All models use random_state=42. Run the same train.py twice — F2 should be identical both times
5 Scope creep from denial reason column
Temptation to switch to multi-class mid-project destabilizes the research loop		 Medium Denial reason is explicitly excluded from program.md as a target. Binary denied vs paid only. Multi-class is a stretch goal for week 6 only. Agent attempts to use denial reason as a target variable in train.py
6 Synthetic data patterns do not generalize
Findings on Kaggle data may not transfer to real claims from actual health systems		 Low CMS Medicare data used as secondary validation in week 6 to cross-check whether key features align with real payment patterns. Features important on Kaggle show no signal in CMS payment gap analysis

6 Project Repository Structure

insurance_autoresearch/
┣━━ prepare.py FROZEN — Data loading, 70/15/15 deterministic split, F2 evaluation function, test set lock
┣━━ train.py EDITABLE — Full modeling pipeline. The only file the agent may modify.
┣━━ run.py FROZEN — Experiment runner: train → evaluate F2 → compare → log to results.tsv
┣━━ program.md — Agent research agenda: what to optimize, what to try, what is forbidden
┣━━ README.md — One-sentence problem, data source, metric, current best F2, setup instructions
┣━━ research_log.md — Dated weekly notes on agent behavior, decisions, and next steps
┣━━ failure_log.md — Crashed experiments, misleading improvements, and lessons learned
┣━━ requirements.txt — Pinned Python dependencies for full reproducibility
┣━━ .gitignore — Excludes data/, results.tsv, performance.png, __pycache__
┣━━ results.tsv GENERATED — Experiment log auto-appended by run.py — one row per trial
┣━━ performance.png GENERATED — F2 score trajectory plot over iterations
┗━━ data/ LOCAL ONLY — Raw Kaggle claims CSV — never pushed to GitHub
Boundary principle: Everything the agent can change is in EDITABLE (only train.py). Everything that measures success is in FROZEN. This separation is what makes every experiment comparable and the research loop trustworthy.
